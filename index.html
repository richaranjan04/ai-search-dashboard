Workstream Analysis
Classification Methodology

All cases tagged as AI Search in the source data were classified into 9 workstreams using a structured two-pass approach:

① Pass 1 — Keyword Matching

Each case subject and full description was scanned against curated keyword lists. Workstreams were applied in priority order — more specific workstreams take precedence over the broad Setup bucket. Keywords matched per workstream:

  • Zing Search
    zing in subject line; zing search anywhere in text
  • Migration & Upgrades
    migrate, migration, upgrade, patch
  • External Content Connectors
    connector, external source, external content, SharePoint connector, crawl, external knowledge
  • Virtual Agent
    virtual agent, Now Assist, NAVA, VA search, VA profile, Now Assist Chatbot, suggested steps generation, agentic workflow
  • Genius Results
    genius result, genius results, Now Assist paired with genius
  • Search Analytics
    search analytics, search signal, sys_search_event, search tracking, search term, sustained reduction
  • Security & Access
    permission, access control, user criteria, visibility, ACL, guest account, restricted knowledge, unauthorized
  • Search Relevancy
    relevancy, ranking, inconsistent results, no results, RAG retrieval, hybrid search, multilingual, boosting, boost action, result improvement rules, semantic, machine learning, recommended action
  • AI Search Setup & Misc
    provision, activate, index event, index queue, search profile, plugin, publish, datasource, AIS status, not enabled, catalog item, dictionary, spell check, self-hosted + broad setup terms

② Pass 2 — SME Validation

Cases not matched in Pass 1 were escalated for subject matter expert review. Each case's full description and resolution notes were evaluated against workstream definitions to ensure accurate placement. Cases outside the AI Search scope (e.g. platform performance, ATF testing, routing requests) were consolidated under Others. After both passes, 0 cases remained unclassified.
